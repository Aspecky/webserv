what is webserv

webserv is an HTTP/1.1 web server built entirely from scratch in C++98, with no external networking libraries allowed. It listens on TCP sockets, multiplexes concurrent connections through a single poll() event loop, and handles the full lifecycle of an HTTP request — from raw bytes on the wire to a served response. The server parses nginx-style configuration files, serves static files, handles redirections and custom error pages, and executes dynamic content through CGI/1.1. It is a 42 School team project designed to demystify the internals of real-world web servers.

architecture layers
#layerdescription
01ConfigParserTokenizes and parses nginx-style .conf files into ServerConfig and LocationConfig structs.
02Socket ServerCreates non-blocking TCP sockets via O_NONBLOCK, binds to ports, starts listening.
03Poll LoopSingle-threaded event loop using poll(). Monitors all fds for POLLIN / POLLOUT.
04ClientPer-connection state machine: READING → WRITING → DONE. Buffers incoming data.
05HttpRequestIncremental feed() parser for request line, headers, body via Content-Length.
06HttpResponseBuilds responses, serves static files, redirections (301/302), custom error pages.
07CGI HandlerForks via execve(), sets REQUEST_METHOD, PATH_INFO, pipes stdout as response.
key concepts
Non-blocking I/O
All sockets use O_NONBLOCK. A slow client can never stall other connections.
State machine
Each Client cycles READING → WRITING → DONE. Poll events drive transitions.
Nginx config syntax
server {} and location {} blocks with root, index, methods, return, upload_store.
Virtual hosting
Multiple server blocks on same port, differentiated by the Host header.
CGI / 1.1
fork() + execve() to launch Python/PHP/Perl. Env vars carry the request context.
Chunked transfer
Transfer-Encoding: chunked for responses where Content-Length is unknown.
FD hygiene
Every fd opened must be tracked and closed. Leaks cause failures under load.
HTTP methods
GET, POST, DELETE — validated per location block. Unsupported → 405.
